1. Images 

  • 1.1 Does each image conveying information have a text alternative?

    Tests and references for criterion 1.1

    1.1.1

    Does each image (tag <img> or tag with the WAI-ARIA attribute role="img") conveying information have a text alternative?

    Test Methodology 1.1.1
    1. Find in the document the images structured using an <img> element or an element with the WAI-ARIA attribute role="img";
    2. For each image, determine whether the image conveys information;
    3. In the case of an <img> element, check that the image has at least one of the following text alternatives:
      • Text associated via the WAI-ARIA attribute aria-labelledby;
      • Content of the WAI-ARIA attribute aria-label;
      • Content of the alt attribute;
      • Content of the title attribute.
    4. In the case of an element with the WAI-ARIA attribute role="img", check that the image has at least one of the following text alternatives:
      • Text associated via the WAI-ARIA attribute aria-labelledby;
      • Content of the WAI-ARIA attribute aria-label.
    5. If at least one text alternative is found, the test is passed.

  • 1.2 Is each decorative image correctly ignored by assistive technologies?

    Tests and references for criterion 1.2

    1.2.1

    Does each image (tag <img>) decorative, without a caption, meet one of these conditions? 

    • The <img> tag has an empty alt attribute (alt="") and is without any other attribute providing a text alternative;
    • The <img> tag has a WAI-ARIA attribute aria-hidden="true" or role="presentation".
    Test Methodology 1.2.1
    1. Find in the document the decorative images without a caption structured using an <img> element;
    2. For each image, check that the image does not have the attributes aria-labelledby, aria-label, or title and that it has:
      • Either an empty alt attribute (alt="");
      • Either a WAI-ARIA attribute aria-hidden="true" or role="presentation".
    3. If this is the case for each image, the test is passed.

    Technical Note

    When an image is associated with a caption, the WCAG technical note recommends always providing a text alternative (cf. criterion 1.9). In this case, criterion 1.2 is not applicable.

    In the case of a decorative vector image (tag <svg>) displayed through a <use href="…"> element within the <svg> tag, test 1.2.4 will also apply to the <svg> element associated via the <use> element.

    A WAI-ARIA attribute role="presentation" can be used for decorative images and non-clickable decorative areas. The "none" role introduced in ARIA 1.1, which is synonymous with the "presentation" role, can also be used. However, it is still preferable to use the "presentation" role until satisfactory support for the "none" role is achieved.

  • 1.3 For each image carrying information that has a text alternative, is this alternative relevant (except in special cases)?

    Tests and references for criterion 1.3

    1.3.1

    Each image (element <img> or element with the WAI-ARIA attribute role="img") carrying information, which has a text alternative, is this alternative relevant (except in special cases)? 

    • If present, the content of the alt attribute is relevant;
    • If present, the content of the title attribute is relevant;
    • If present, the content of the WAI-ARIA aria-label attribute is relevant;
    • If present, the passage of text associated via the WAI-ARIA aria-labelledby attribute is relevant.
    Test Methodology 1.3.1
    1. Find in the document the images structured using an <img> element (or an element with the WAI-ARIA role="img" attribute) with a text alternative;
    2. For each image, check that the text alternative is relevant;
    3. If this is the case for each image, the test is passed.

    1.3.9

    For each image carrying information and having a text alternative, is the text alternative short and concise (except in special cases)? 

    Test Methodology 1.3.9
    1. Find in the document the images with a text alternative;
    2. For each image, check that the text alternative is short and concise;
    3. If this is the case for each image, the test is passed.

    Special cases

    There is special handling for cases where the image is used as a CAPTCHA or as an test image. In this situation, where it is not possible to provide a relevant alternative without compromising the purpose of the CAPTCHA or test, the criterion is not applicable.

    Note: the CAPTCHA and test image cases are specifically addressed by criterion 1.4.

  • 1.6 Does each image bearing information have, if necessary, a detailed description?

    Tests and references for criterion 1.6

    1.6.1

    Does each image (tag <img>) bearing information, which requires a detailed description, meet one of these conditions? 

    • There is a longdesc attribute providing the address (URL) of a page or location within the page containing the detailed description;
    • There is a text alternative containing the reference to an adjacent detailed description;
    • There is an adjacent link or button leading to the detailed description.
    Test Methodology 1.6.1
    1. Find images in the document structured using an <img> tag (or an element with the WAI-ARIA attribute role="img") bearing information that requires a detailed description;
    2. For each image, check that one of the following exists:
      • Either a longdesc attribute giving the address (URL) of a page or location within the page containing the detailed description;
      • Either a text alternative containing the reference to an adjacent detailed description;
      • Either an adjacent link or button leading to the detailed description.
    3. If this is the case for each image, the test is passed.

    1.6.10

    Does each tag with a WAI-ARIA attribute role="img" bearing information, which requires a detailed description, meet one of these conditions? 

    • There is a WAI-ARIA attribute aria-label containing the text alternative and a reference to an adjacent detailed description;
    • There is a WAI-ARIA attribute aria-labelledby associating a text passage acting as a text alternative and another acting as the detailed description;
    • There is a WAI-ARIA attribute aria-describedby associating a text passage acting as the detailed description;
    • There is an adjacent link or button leading to the detailed description.
    Test Methodology 1.6.10
    1. Find elements with a WAI-ARIA attribute role="img" bearing information that requires a detailed description;
    2. For each element role="img", check that one of the following exists:
      • Either a WAI-ARIA attribute aria-label containing the text alternative and a reference to an adjacent detailed description;
      • Either a WAI-ARIA attribute aria-labelledby associating a text passage acting as the text alternative and another acting as the detailed description;
      • Either a WAI-ARIA attribute aria-describedby associating a text passage acting as the detailed description;
      • Either an adjacent link or button leading to the detailed description.
    3. If this is the case for each element role="img", the test is passed.

    Technical Notes

    In the case of SVG, the lack of support for the <title> and <desc> elements by assistive technologies creates a challenge in implementing the text alternative for the image and its detailed description. In this case, it is recommended to use the WAI-ARIA attribute aria-label to implement both the short text alternative and the reference to the adjacent detailed description or the WAI-ARIA attribute aria-labelledby to associate the text passages acting as both the short alternative and the detailed description.

    The use of the WAI-ARIA attribute aria-describedby is not recommended to link an image (<img>, <object>, <embed>, <canvas>) to its detailed description, due to lack of support by assistive technologies. However, when used, the attribute must necessarily refer to the id of the area containing the detailed description.

    The text alternative for an image can be used for cases like banners, but a detailed description should be used if the image links to an action or provides additional information on the content.

  • 1.8 Each text image carrying information, in the absence of a replacement mechanism, should if possible be replaced by styled text. Is this rule respected (except for special cases)?

    Tests and references for criterion 1.8

    1.8.1

    Each text image (tag <img> or with the WAI-ARIA attribute role="img") carrying information, in the absence of a replacement mechanism, should if possible be replaced by styled text. Is this rule respected (except for special cases)?

    Test Methodology 1.8.1
    1. Find in the document the text images structured with an <img> element (or an element with the WAI-ARIA role="img" attribute);
    2. For each image, check that:
      • Either there is a replacement mechanism;
      • Or the image contains text that uses a graphical effect that cannot be reproduced in CSS.
    3. If this is the case for each image, the test is passed.

    Special cases

    For this criterion, there is special case handling when the text is part of a logo, a commercial name, a CAPTCHA, an image-test, or an image whose graphic accuracy would be considered essential to the correct transmission of the information carried by the image. In these situations, the criterion does not apply to these elements.

    Technical notes

    Text in vector images is real text and is therefore not subject to this criterion.

  • 1.9 Is each image caption correctly linked to the corresponding image, if necessary?

    Tests and references for criterion 1.9

    1.9.1

    For each image with a caption (element <img>, <input> with attribute type="image" or with a WAI-ARIA attribute role="img" associated with an adjacent caption), does it, if necessary, meet these conditions? 

    • The image (element <img>, <input> with attribute type="image" or with WAI-ARIA attribute role="img") and its adjacent caption are contained in a <figure> tag;
    • The <figure> tag has a WAI-ARIA attribute role="figure" or role="group";
    • The <figure> tag has a WAI-ARIA attribute aria-label whose content is identical to the content of the caption;
    • The caption is contained in a <figcaption> tag.
    Test Methodology 1.9.1
    1. Find in the document images with captions structured using the <img> element, the <input> element with type="image", or an element with the WAI-ARIA attribute role="img";
    2. For each image, check that:
      • The image and its caption are contained in a <figure> tag;
      • The <figure> tag has a WAI-ARIA property role="figure" or role="group";
      • The <figure> tag has a WAI-ARIA attribute aria-label whose content is identical to the caption content;
      • The caption is contained in a <figcaption> tag.
    3. If this is the case for each image, the test is passed.

    Technical Note

    The implementation of a WAI-ARIA attribute role="group" or role="figure" on the parent <figure> element is intended to compensate for the current lack of support for <figure> elements by assistive technologies. Using a <figcaption> element to associate a caption with an image requires, at a minimum, the use of a WAI-ARIA aria-label attribute on the parent <figure> element with content identical to the content of the <figcaption> element. To ensure optimal support, explicit association between the image's alternative text and the <figcaption> content may also be made, for example:

    <img src="image.png" alt="Photo: sunset" /><figcaption>Photo: credit xxx</figcaption>

    The WAI-ARIA attributes aria-labelledby and aria-describedby cannot currently be used due to lack of support by assistive technologies.

    Note: Captioned images must also comply with criterion 1.1 and criterion 1.3 regarding images carrying information.

    Tests and references for criterion 3.1

    3.1.1

    For each word or set of words where color conveys information, the information must not be conveyed solely by color. Is this rule respected?

  • 3.2 On each web page, is the contrast between the text color and its background color sufficiently high (except for special cases)?

    Tests and references for criterion 3.2

    3.2.1

    On each web page, does text and text in images without bold effect of a displayed size smaller than 24px meet one of these conditions (except for special cases)?

    • The contrast ratio between the text and its background is at least 4.5:1;
    • A mechanism allows the user to display the text with a contrast ratio of at least 4.5:1.
    Test Methodology 3.2.1
    1. Identify text and text in images without bold effect of a displayed size smaller than 24px that may present contrast issues;
    2. For each of these texts, check that:
      • Either the contrast ratio between the text and its background is at least 4.5:1;
      • Or a mechanism allows the user to display the text with a contrast ratio of at least 4.5:1.
    3. If this is true for each text, the test is passed.

    3.2.2

    On each web page, does text and bold text in images with a displayed size smaller than 18.5px meet one of these conditions (except for special cases)?

    • The contrast ratio between the text and its background is at least 4.5:1;
    • A mechanism allows the user to display the text with a contrast ratio of at least 4.5:1.
    Test Methodology 3.2.2
    1. Identify bold text and text in images with a displayed size smaller than 18.5px that may present contrast issues;
    2. For each of these texts, check that:
      • Either the contrast ratio between the text and its background is at least 4.5:1;
      • Or a mechanism allows the user to display the text with a contrast ratio of at least 4.5:1.

    3.2.3

      • Either the contrast ratio between the text and its background is at least 3:1;

    3.2.4

    On each web page, does bold text and text in images with a displayed size greater than or equal to 18.5px meet one of these conditions (except for special cases)?

    • The contrast ratio between the text and its background is at least 3:1;
    • A mechanism allows the user to display the text with a contrast ratio of at least 3:1.

    1. Identify the mechanisms that allow the display of a compliant contrast ratio;
    2. For each of these mechanisms, verify that the contrast ratio between the text and the background color is sufficiently high;

    • The text is part of an image conveying information but the text itself does not provide any essential information;

  • 3.3 In each web page, are the colors used in the interface components or the graphic elements carrying information sufficiently contrasted (excluding special cases)?

    Tests and references for criterion 3.3

    3.3.1

    In each web page, does the contrast ratio between the colors of an interface component in its various states and the adjacent background color meet one of these conditions (excluding special cases)? 

    • The contrast ratio is at least 3:1;
    • An mechanism allows a contrast ratio of at least 3:1.
    Test Methodology 3.3.1
    1. Find in the document the interface components that may pose contrast issues;
    2. For each of these components, verify that:
      • Either the contrast ratio between the component's colors in its various states and the adjacent background color is at least 3:1;
      • Or a mechanism allows the user to display the component with a contrast ratio of at least 3:1.
    3. If this is the case for each component, the test is passed.

    3.3.2

    In each web page, does the contrast ratio of the different colors composing a graphic element, when necessary for its understanding, and the adjacent background color meet one of these conditions (excluding special cases)? 

    • The contrast ratio is at least 3:1;
    • An mechanism allows a contrast ratio of at least 3:1.
    Test Methodology 3.3.2
    1. Find in the document the graphic elements that may pose contrast issues;
    2. For each of these elements, verify that:
      • Either the contrast ratio between the colors of the graphic element necessary for its understanding and the adjacent background color is at least 3:1;
      • Or a mechanism allows the user to display the graphic element with a contrast ratio of at least 3:1.
    3. If this is the case for each component, the test is passed.

    3.3.3

    In each web page, does the contrast ratio of the different adjacent colors between themselves in a graphic element, when necessary for its understanding, meet one of these conditions (excluding special cases)?

    • The contrast ratio is at least 3:1;
    • An mechanism allows a contrast ratio of at least 3:1.
    Test Methodology 3.3.3
    1. Find in the document the graphic elements that may pose contrast issues;
    2. For each of these elements, verify that:
      • Either the contrast ratio of the different adjacent colors in the graphic element between themselves, when necessary for its understanding, is at least 3:1;
      • Or a mechanism allows the user to display the graphic element with a contrast ratio of at least 3:1.
    3. If this is the case for each graphic element, the test is passed.

    3.3.4

    In the mechanism that allows the display of a compliant contrast ratio, are the colors of the component or the graphic elements conveying information that make it up, sufficiently contrasted? 

    Test Methodology 3.3.4
    1. Find in the document the mechanisms that allow the display of a compliant contrast ratio;
    2. For each of these mechanisms, check that the contrast ratio between the colors of the component or the graphic elements conveying information that make it up is sufficiently high;
    3. If this is the case for each mechanism, the test is passed.

    Note: the criterion does not apply in these situations:

    • Inactive interface component (for example, a button with a disabled attribute) on which no action is possible;
    • Interface component for which the appearance is managed by the browser's native styles without any modification by the author (for example, the native focus style in Chrome or Firefox);
    • Interface component for which color is not necessary to identify the component or its state (for example, a group of links serving as navigation where the position in the page, size, and color of the text help to understand that they are links even if the underline color of the links against a white background does not meet the 3:1 ratio, while the text itself has a 4.5:1 ratio);
    • Graphic element or parts of a graphic element that do not convey information or have an alternative (long description, identical information visible on the page);
    • Graphic element or parts of a graphic element that are part of a logo or the brand name of an organization or company;
    • Graphic element or parts of a graphic element whose presentation is essential to the conveyed information (for example, flags, logos, photos of people or scenes, screenshots, medical diagrams, heat maps);
    • Dynamic graphic element or parts of a graphic element where the contrast on hover/focus is sufficient.

    Special cases

    The following cases are not applicable for this criterion:

    • Inactive interface component (e.g., a button with a disabled attribute) on which no action is possible;
    • Interface component for which the appearance is managed by the native browser styles without any modification by the author (e.g., native focus style in Chrome or Firefox);
    • Interface component for which color is not necessary to identify the component or its state (e.g., a group of links acting as navigation where the position on the page, the size, and color of the text allow the user to understand that they are links even if the underline color of the links with a white background does not have a 3:1 contrast ratio and the text itself has a 4.5:1 contrast ratio);
    • Graphic element or parts of a graphic element that do not convey information or have an alternative (long description, identical information visible on the page);
    • Graphic element or parts of a graphic element that are part of a logo or a brand name of an organization or company;
    • Graphic element or parts of a graphic element whose presentation is essential to the information conveyed (e.g., flags, logos, photos of people or scenes, screenshots, medical diagrams, heatmaps);
    • Graphic element or parts of a graphic element whose hover/focus contrast is sufficient.

5. Table 

  • 5.1 Does each complex data table have a summary?

    Tests and references for criterion 5.1

    5.1.1

    For each complex data table, is a summary available?

    Test Methodology 5.1.1
    1. Locate in the document the complex data tables (data table - <table> element or an element with a WAI-ARIA role="table" attribute - containing headers that are not only spread across the first row and/or first column of the grid, or whose scope does not apply to the entire column or row);
    2. For each complex data table, verify that a text passage that allows understanding the nature and structure of the table is present:
      • Either in the <caption> element;
      • Or in the summary attribute of the <table> element (in HTML and XHTML versions prior to HTML 5);
      • Or in a text passage related to the table with the aria-describedby attribute.
    3. If this is the case for each complex data table, the test is validated.

    Technical Notes

    The HTML specification provides several methods to link a summary to a table (table linked to a text passage with the aria-describedby attribute, table grouped in a figure element with a summary in a figcaption or p element, summary present in a details element contained in the caption element). These methods do not currently have sufficient support for use.

  • 5.2 For each complex data table with a summary, is it relevant?

    Tests and references for criterion 5.2

    5.2.1

    For each complex data table with a summary, is it relevant? 

    Test Methodology 5.2.1
    1. Locate the summaries of complex data tables (as determined by test 5.1.1);

    Tests and references for criterion 6.1

    6.1.2

    Chaque lien image vérifie-t-il une de ces conditions (hors cas particuliers) ?

    • L'intitulé de lien additionné au contexte du lien permet d'en comprendre la fonction et la destination.
    Test Methodology 6.1.2

      • Or the link context.

    6.1.4

    Does each SVG link meet one of these conditions (except in special cases)? 

    • The link title alone allows understanding of its function and destination;
    • The link title combined with the link context allows understanding of its function and destination.
    Test Methodology 6.1.4
    1. Identify SVG links in the document (an <svg> element that contains an <a> element with a xlink:href attribute (SVG 1.1) or href attribute (SVG 2));
    2. For each SVG link, verify that what allows understanding of its function and destination is:
      • Either the link title alone, provided by the accessible name of the <svg> element (usually resolved from the content of a <text> element);
      • Or the link context.
    3. If this is true for each SVG link, the test is validated.

    6.1.5

    For each link with a visible label, does the accessible name of the link contain at least the visible label (except in special cases)? 

    Test Methodology 6.1.5
    1. Identify in the document all non-SVG links whose content is provided both by a visible label and by either a title attribute, an aria-label attribute, or an aria-labelledby attribute;
    2. For each link, verify that the content of the title attribute, aria-label attribute, or aria-labelledby attribute contains the visible label;
    3. If this is true for each link, the test is validated for non-SVG links.
    4. Identify in the document all SVG links whose content is provided both by a visible label and by either an aria-labelledby attribute, an aria-label attribute, a <title> element (direct child of the <svg> element), an x-link:title attribute (SVG 1.1), or one or more <text> elements;
    5. For each SVG link, verify that the content of the aria-labelledby attribute, aria-label attribute, <title> element, x-link:title attribute, or one or more <text> elements contains the visible label;
    6. If this is true for each SVG link, the test is validated for SVG links.
    7. If the test is validated for both non-SVG and SVG links, the test is globally validated.

    Note: Regarding the determination of the accessible name, there are two special cases and one exception related to mathematical expressions:

    • Punctuation and capital letters present in the visible label text may be ignored in the accessible name without consequence.
    • If the visible label text serves as a symbol, it should not be interpreted literally in the accessible name. The name should express the function conveyed by the symbol (for example, "B" in a text editor should have an accessible name such as "Bold," while the ">" symbol, depending on the context, may mean "Next" or "Play video"). However, mathematical symbols are an exception (see below).
    • If the visible label represents a mathematical expression, the mathematical symbols may be used literally as the accessible name label (e.g., "A>B"). The user is responsible for determining how to vocalize the expression based on their knowledge of their speech input software ("A greater than B" or "A is greater than B").

    Special Cases

    There is a special case management for tests 6.1.1, 6.1.2, 6.1.3, and 6.1.4 when the link is ambiguous for everyone. In this situation, where it is not possible to make the link explicit in its context, the criterion is not applicable.

    There is a special case management for test 6.1.5 when:

    • Punctuation and capital letters are present in the visible label: they can be ignored in the accessible name without consequence;
    • The text of the visible label serves as a symbol: the text should not be interpreted literally in the accessible name. The name should express the function conveyed by the symbol (for example, "B" in a text editor should have the accessible name "Bold", the ">" symbol, depending on the context, could mean "Next" or "Play video"). However, mathematical symbols are an exception (see the note below).

    Note: If the visible label represents a mathematical expression, the mathematical symbols may be used literally as the accessible name (e.g., "A>B"). It is up to the user to interpret the expression according to their knowledge of how their voice input software works ("A greater than B" or "A superior to B").

    Technical Notes

    When the visible label is supplemented by another expression in the accessible name:

    • WCAG emphasizes placing the visible label at the beginning of the accessible name, though it does not require exclusivity of this placement;
    • WCAG considers it a failure if the visible label's character string does not match exactly within the accessible name.

    For example, considering the visible label "Order now” supplemented in the accessible name by the expression "product X”, the following cases may occur:

    • "Order now product X” is valid (best practice);
    • "Product X: order now” is valid;
    • "Order product X now” is invalid.

Best Practises

If you have successfully passed all the required tests, your document is accessible! However, complying with these standards does not necessarily mean that the document is truly accessible to the widest audience.

A commonly cited example is font size. To date, there is no specific criterion defining a minimum regulatory size for text. While the text must be zoomable, there is no clear indication of the minimum required size.

This is where recommendations from a range of specialists, developers, and people with visual or cognitive impairments come into play.

These recommendations, often based on common sense and sometimes unexpected, are just as important as WCAG guidelines in practice.

They represent a small set of knowledge to remember and habits to adopt in order to create the most accessible document (web, email, etc.) possible.

  • 14.1 Avoid Long URLs

    Details

    14.1.1

    Refrain from including lengthy URLs within the main content of your email.

    Rather, opt for meaningful link text that clearly conveys the destination or purpose of the link. This enhances readability and keeps the email visually tidy for all recipients.

    Example: Instead of saying, "Check out this on Google: http://www.google.com" use "check out this on Google" and embed the link under "Google".

  • 14.2 Emojis and Emoticons

    Details

    14.2.1

    Emojis and Emoticons

    Use emojis sparingly. Avoid using text-based emoticons. Instead, use plain language or emojis to keep the message clear for all users, including those using screen readers. Example: instead of "Great job! :)", use "Great job! 😊"

    Using only an emoji (and no text) increases the cognitive load for people using assistive technologies – because they rely solely on the alt text to interpret the emoji.

    Misunderstandings can also lead to people feeling excluded, which goes against the goal of accessibility.

    Emojis should be placed at the end of your message, social caption, or sentence.

    Avoid long emoji strings

    Emojis should only be used to add to the context of your text – never as a replacement.

    Here are two ways to correctly use emojis for screen readers:

    Adding Alt :

    We <img src="emoticon-heart.png" width="22" height="22" alt="love" style="display: inline; margin: 0; padding: 0; vertical-align: -5px;" border="0" /> email!

    Or using Wai-Aria :

    We <span role="img" aria-label="love">❤️</span> email!

  • 14.3 Marking Any Links to PDFs

    details

    14.3.1

    Marking Any Links to PDFs

    Label links to PDF files as PDFs. This informs users what to expect when clicking on the link.

    Example: Instead of "Download the file," use "Download the PDF file" or "Download File (PDF)".

  • 14.4 Avoid Underlining Non-Link Text

    Details

    14.5.1

    Avoid Underlining Non-Link Text

    Refrain from underlining text that isn't a hyperlink to prevent confusing users. Underlines should be reserved exclusively for links, making them easily recognizable.

    Use underlines solely to highlight clickable links.

  • 14.5 Text and Fonts

    Details

    14.5.1

    Text and Fonts

    Choose a simple, legible font for emails. Avoid complex or hard-to-read fonts, and ensure the text size is at least 12 points. Align the text to the left.

    It is usually advised to steer clear of serif fonts (those with small lines at the ends of letters) as they can hinder readability for people with dyslexia by making it more difficult to distinguish individual letters.

    Opt for sans-serif font, and a standard one like Arial or Times New Roman.

    14.5.2

    Centering Text

    Avoid centering text if it spans more than a few lines. Keep in mind that a small amount of centered text on desktop could turn into multiple lines on mobile, making it harder to read.

    14.5.3

    Justifying Text

    Justified text is spaced so that both the left and right edges of the paragraph are aligned with the page margins, creating a block-like appearance.

    The way text is laid out affects its readability. Justifying text can make it harder for some people to follow the text from line to line, especially those with tracking issues.

    This becomes a significant issue for users with assistive technologies (AT). Many people with vision impairments rely on screen magnifiers or zoom in on web content. If the text is justified, these users may encounter large gaps between words or words that run together, creating confusion.

    14.5.4

    Avoid Black Text on White Background

    A significant number of users with dyslexia are particularly sensitive to the high contrast created by these colors, which can cause words to appear distorted or blurred.

    To improve readability, consider using a softer background color, such as light gray or beige, instead of pure white. Additionally, opting for dark gray text rather than pure black can help reduce glare and enhance overall legibility.

    14.5.5

    Heading Style

    As the display of headings (margins, weight, etc.) can vary across different mailboxes, a best practice is to double the style of the td directly in the hn tag. For example :

    <td style="font-family:Arial,sans-serif;font-size:14px;color:#000000;padding:10px">
    <h1 style="margin:0;font-family:Arial,sans-serif;font-size:14px;color:#000000;">Lorem ipsum</h1>
    </td>

    La poste.net adds a background color and a border to level 2 and 3 titles.

    To ensure consistency, always add:

    background:none;border:0

    to h2 and h3 tags to avoid differences in rendering in this mail client.

    14.5.6

    Avoid Using All Caps

    Steer clear of using all caps, as screen readers may interpret them as separate letters. Opt for regular or title case to improve readability.

    Example: Instead of writing FINALLY, write finally.

    Use CSS text-transform: uppercase

    So, instead of writing:

    <p>BONJOUR</p>

    Use:

    <p style="text-transform:uppercase">bonjour</p>

  • 14.6 Use Descriptive Anchor Link Text

    Details

    14.6.1

    Use Descriptive Anchor Link Text

    The recommendations from the Section 508 guidelines are very clear on this subject:

    "Ensure link text is meaningful and descriptive, indicating the purpose or destination of the link. Avoid using vague terms like "click here" or "read more."" Example: Use "Download the annual report" instead of "Click here."" The link text should make sense out of context.

    If the editorial part cannot be edited, text can be displayed only for screen readers.

    CSS properties like absolute positioning or clip are not really supported in email. There are several hacks available that we will present in section 14.12 (see below).

  • 14.7 Whitespace and Layout

    Details

    14.7.1

    Whitespace and Layout

    Use paragraph spacing and layout tools to create a clean presentation. Avoid using tabs, multiple spaces, or carriage returns to create whitespace.

    Use appropriate line spacing and margins to separate sections of text.

    14.7.2

    Line Break in text

    For a line break, use the margin on paragraph tags, rather than a <br/>. The height of the line break induced by a <br/> varies from one mailbox to another. It should therefore be avoided, as much as possible.

    <p style="margin:0 0 10px">

    From an accessibility perspective, the line break is a visual indicator, not a semantic one. Depending on screen readers, they may pause on the br. If the line break is purely visual, one way to avoid it is to use role=text.

    <span role="text">The<br>End</span>

    Or

    <br aria-hidden="true" />

    Unfortunately, depending on the screen reader and the email client used, these techniques may have no effect.

    14.7.3

    Line Breaks in Headings

    According to Litmus, when splitting a headline in two parts with a <br/> tag, the screen reader could read the heading twice.

    A solution would be to wrap the line you want to break in a <span> with style="display:block;" applied to it, which breaks it into a new line.

    Then, you can add a responsive class="inline" to it so the copy flows inline again once it hits below your mobile breakpoint. Then add an   before the <span> to force a separation between the words for the screen reader, and an optional &nbsp; at the end of the copy to prevent orphans. See code examples below:

    @media only screen and (max-width: 640px) {
    .inline { display:inline !important; }
    }

    In reality, if you navigate with the arrows, but not only that, as soon as a title spans multiple lines, NVDA (to name just one) will read "heading level n" as many times as there are lines. This bug has been reported in many discussions, and there is no solution to this problem.

  • 14.8 Underline

    Details

    14.8.1

    Underline

    The fundamental visual indicator of a link is the underline, which helps distinguish it from the surrounding text.

    While this is effective, there is an issue with the underlining itself. Each letter has a baseline, which can be thought of as the line on which the letter sits. Certain letters, such as g, p, q, y, and j, extend below this baseline, giving them "dangling feet." When these letters are underlined, the line crosses over these lower parts, making them harder to read.

    Instead of text-decoration:underline, use border-bottom.

    a.border {
    line-height: 1.5em;
    text-decoration: none;
    border-bottom: 1px solid blue;
    }

    text-decoration-skip is still an experimental technology.

    text-decoration-skip-ink is now widely available.

  • 14.9 List

    Details

    14.9.1

    Regarding list margins:

    Outlook transforms any list into a paragraph, as soon as you touch the lateral margin (margin-left). The rendering will be that of a list, but in the code and on Outlook, the list will be made up of paragraphs, so we lose accessibility.

    In the ul or li tags, you must therefore indicate a vertical margin only.

    To visually adjust the list while remaining accessible, there are different techniques to control the positioning of the list or reduce the margins between the bullet points and the text.

    • reduce the margin between the bullet points and the text.
    • move the list horizontally by encapsulating the list in a div or use a simple layout trick.
    • create a fake list using a paragraph and role="list"

  • 14.10 Abbreviations

    Details

    14.10.1

    From an accessibility point of view, it is recommended to surround abbreviations with an <abbr> tag with a title to indicate the long version (Latin extension) of the word.

    <abbr title="World Wide Web">WWW</abbr>

    If you use the <abbr> tag, it is also recommended to use the CSS property speak:spell-out to announce that an element must be spelled out.

    The abbr tag semantically indicates that it is an abbreviation or an acronym, but it has no effect on overall accessibility.

    This solution is not optimal, because the majority of screen readers do not read the title attribute.

    It is therefore recommended to use an aria-label.

    <span aria-label="Mister">Mr</span>

    Since screen readers on Outlook do not read WAI-ARIA, one solution is to use hidden text that is only visible to screen readers. This technique is described below in section 14.12.

  • 14.11 Background image in Outlook

    Details

    14.11.1

    To display a background image in Outlook (Windows desktop version), the most common method is to use VML code. However, it is important to note that the default VML is not accessible. In Outlook, text contained within VML tags like v:rect, v:shape or v:roundrect is not interpreted by screen readers (such as text on an image, VML buttons, etc.).

    After many hours of testing and thanks to the preliminary research of Mark Robbins, I discovered a new approach in 2023 that allows for absolute positioning in Outlook, particularly by utilizing mso-specific properties.

    This technique enables, among other things, displaying text over an image while maintaining accessibility.

    For more information, I invite you to visit my GitHub page and my "mso-faux-absolute generator".

    Forcing screen readers on Outlook to read text over an image

    I provide detailed explanations of this technique in section 14.12, but in short, since text contained within v:rect, v:shape, or v:roundrect tags is not typically interpreted by screen readers, we can use a small hack.

    Here is a simple example, using the backgrounds.cm code


    <div background="https://picsum.photos/680/115" bgcolor="#7bceeb" height="115" valign="top" style="background-image:url('https://picsum.photos/680/115');background-size:cover;height:115px">
    <!--[if gte mso 9]>
    <v:rect role="presentation" xmlns:v="urn:schemas-microsoft-com:vml" fill="true" stroke="false" style="width:512pt;height:86.25pt;">
    <v:fill type="frame" src="https://picsum.photos/680/115" color="#7bceeb" />
    <v:textbox inset="0,0,0,0">
    <![endif]-->
    <div style="margin:0">
    <h1 style="margin:0">mon titre</h1>
    <p style="margin:0">texte</p>
    </div>
    <!--[if gte mso 9]>
    </v:textbox>
    </v:rect>
    <![endif]-->
    </div>
    <!--[if gte mso 9]>
    <table role="presentation"><tr><td style="height:0; width:0;font-size:1.0pt;mso-font-width:1%;mso-style-textfill-type: none;">
    <h1 style="font-size:1.0pt;mso-font-width:1%;mso-style-textfill-type: none;">Title for screen reader</h1>
    <span style="font-size:1.0pt;mso-font-width:1%;mso-style-textfill-type: none;">text for screen reader</span>
    </td></tr></table>
    <tr><td align="center" style="font-family: Verdana, Arial,sans-serif; font-size:14px; mso-line-height-rule: exactly;line-height: 21px; color: #000000;background-color:#ffffff;padding35px 20px;">
    <![endif]-->

    We use a standard VML code that a screen reader will ignore in Outlook, and we add hidden text outside the VML tags that the screen reader will read. This approach ensures that screen reader users can access the content, even though the VML itself is not accessible by default.

    Unlike VML buttons, in the case of background images, the hidden text hack doesn't always work perfectly. The VML component tends to block the cursor or cause glitches. While the reading experience is less fluid, using arrow keys, for example, the screen reader can still correctly announce the hidden content, including headings, text, and links. This is a simple way to make your VML blocks much more accessible!

    Additional testing is needed, and your help in this area would be greatly appreciated!

  • 14.12 Making your link texts accessible, with or without VML.

    Details

    14.11.1

    On a majority of modern email clients (hotmail, gmail, Apple Mail etc.), an aria-label on a link allows you to add additional text for screen readers :

    <a href="https://google.com" aria-label="read more about this page">read more</a>

    The screen reader (Narrator, NVDA on Chrome on Windows) will read "read more about this page". It will read the aria-label, ignoring the link text.

    A hidden text HTML version also gives the same result:

    <a href="https://google.com"&gt;read more<span style="display:inline-block; max-height:0; max-width:0;mso-font-width:0%;mso-style-textfill-type: none; white-space: nowrap;"><span style="max-height:1px; max-width:1px; display:inline-block; overflow:hidden; font-size:1px;color:rgba(0,0,0,0);text-indent:9px;">about this page</span></span></a>

    A detailed description of this technique can be found on Mark's page.

    With this HTML hack, the screen reader will read "read more about this page" on most email clients, including Outlook.

    You could place this HTML hack outside of the link, but the screen reader will specify that you are out of the link: "read more [out of link] about this page".

    In Outlook, WAI-ARIA attributes are not read. To create a version of the hack specifically for Outlook, keeping only the relevant CSS properties, we can simplify the code above and target just the necessary styles for Outlook's unique rendering. For example :

    <a href="https://google.com"&gt;read more<span style="height:0; width:0; font-size:0px;text-decoration: none;mso-font-width:0%;mso-style-textfill-type: none;">about this page</span></a>

    However, it is important to note one thing about this code: Upon receipt, Outlook transforms:

    height:0; width:0; font-size:0px;mso-font-width:0%;mso-style-textfill-type: none;

    into:

    font-size:1.0pt;mso-bidi-font-size:24.0pt;mso-fareast-font-family:"Times New Roman";color:black;mso-color-alt:windowtext;mso-font-width:1%;mso-style-textfill-type:none;mso-style-textfill-fill-gradientfill-shadetype:linear;mso-style-textfill-fill-gradientfill-shade-linearshade-angle:0;mso-style-textfill-fill-gradientfill-shade-linearshade-fscaled:no

    Whether you use font-size:1px, font-size:0, font-size:1pt, or mso-ansi-font-size:0, mso-ansi-font-size:1px, or mso-ansi-font-size:1pt, Outlook will convert your code to font-size:1.0pt. This seems to be the minimum size accepted.

    It is also important to note that the width and height properties are ignored by Outlook when applied to a span element. As a result, these properties can be omitted.

    So for Outlook, we can shorten our code for screen readers like so :

    font-size:1.0pt;mso-font-width:1%;mso-style-textfill-type: none;

    The rest of the mso-properties are automatically injected by Outlook, so there's no need to code them.

    The visual rendering will be exactly the same, and the screen reader on Outlook will read "read more about this page".

    Summary:

    1. On a classic HTML link, to add context to the link text, you can use the aria-label attribute, but it will not be read by all email clients.
    2. You can therefor add hidden text for screen readers, with a hack that will be read in almost all email clients, including Outlook, Yahoo etc. :
      <a href="https://google.com"&gt;read more<span style="display:inline-block; max-height:0; max-width:0;mso-font-width:0%;mso-style-textfill-type: none; white-space: nowrap;"><span style="max-height:1px; max-width:1px; display:inline-block; overflow:hidden; font-size:1px;color:rgba(0,0,0,0);text-indent:9px;">about this page</span></span></a>
    3. For a solution that only targets the Outlook mailbox, we can simplify the code as follows:
      <a href="https://google.com" aria-label="read more about this page">read more<!--[if mso]><span style="font-size:1.0pt;mso-font-width:1%;mso-style-textfill-type: none;">about this page</span><![endif]--></a>
      Here, we use a small hack for Outlook, and a simple aria-label for all other email clients.
    4. Here is an example with a rounded button, using VML code (v:roundrect).
      <!--[if mso]>
      <v:roundrect xmlns:v="urn:schemas-microsoft-com:vml" xmlns:w="urn:schemas-microsoft-com:office:word" href="https://google.com" style="height:40px;v-text-anchor:middle;width:200px;" arcsize="10%" strokecolor="#1e3650" fillcolor="#556270"><center style="color:#ffffff;font-family:sans-serif;font-size:13px;font-weight:bold">Read more</center></v:roundrect>
      <a href="http://google.com" style="font-size:1.0pt;mso-font-width:1%;mso-style-textfill-type: none;">about this page</a><![endif]--><a href="http://google.com" style="background-color:#556270;border:1px solid #1e3650;border-radius:4px;color:#ffffff;display:inline-block;font-family:sans-serif;font-size:13px;font-weight:bold;line-height:40px;text-align:center;text-decoration:none;width:200px;-webkit-text-size-adjust:none;mso-hide:all;" aria-label="read more about this page">Read more</a>
      </div>

    Using this technique, your VML button will be more accessible on Outlook: it will pause briefly on the real VML button, then immediately proceed to the hidden link by reading "read more about this page". This invisible button will be clickable for screen reader users. In other email clients, the screen reader will read the normal HTML link, taking the aria-label into account: "read more about this page".

    Namely to choose between aria-label or hidden text : aria-label text doesn't get picked up by find in page search, but hidden text does.

    Things are [changing for browsers](https://adrianroselli.com/2019/11/aria-label-does-not-translate.html), but it is possible that the text contained in an aria-label may not be translatable.
